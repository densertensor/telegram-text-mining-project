# Протокол воспроизводимости

Динамика тем и тональности в «патриотических» Telegram-каналах: корпус **президента** (v2) и корпус **исполнителей** ограничений интернета, анализ «до/после» отсечки 2026-01-16, итоговый пакет `results/`.

Этот документ – полный протокол воспроизведения. Все команды собраны в одном скрипте **`reproduce.sh`**, корректность результата подтверждается автопроверкой **`verify_reproduction.py`** против эталона **`reproduce_expected.json`** (28 проверок).

## Быстрый старт

```bash
cd <корень репозитория>

./reproduce.sh check    # 1. окружение и входные данные на месте?
./reproduce.sh verify   # 2. сверить ТЕКУЩИЕ артефакты с эталоном (≈3 сек)
./reproduce.sh all      # 3. полный перепрогон конвейера + сверка (GPU, ~6–10 ч)
```

`verify` печатает таблицу PASS/WARN/FAIL и возвращает код 0 только если все обязательные проверки сошлись. Это и есть подтверждение репродукции.

| `topic_model_outputs_dynamic_putin_v2_user_bge/` | BERTopic + тональность по 161 599 док. (окно 2022-02-24…2026-04-24), графики | topics, sentiment |

| `results/` | итоговые рисунки 01–06, данные, README | package |

## Окружение

Два conda-окружения (зависимости заморожены в `env/`):

| Env | Python | Зачем | Freeze |
|---|---|---|---|
| `topic-sentiment` | 3.10 | BERTopic, sentence-transformers, ансамбль тональности, графики | `env/requirements_topic-sentiment.txt` |
| `geracl` | 3.11 | zero-shot GeRaCl (нужен `transformers>=4.49`, конфликтует с основным env) | `env/requirements_geracl.txt` |

Модели (HF, кэшированы локально; прогоны шли с `HF_HUB_OFFLINE=1`): эмбеддинги `deepvk/USER-bge-m3`; тональность `cardiffnlp/twitter-xlm-roberta-base-sentiment`, `seara/rubert-base-cased-russian-sentiment`, `deepvk/GeRaCl-USER2-base` (локальный чекпойнт, путь задаётся переменной `GERACL_MODEL`).

Пути к интерпретаторам/модели переопределяются переменными `PY`, `GERACL_PY`, `GERACL_MODEL`, список GPU – `GPUS` (по умолчанию все видимые).

## Входные данные

Единственный вход конвейера и единственный распространяемый источник данных – SQLite-база `patriot_channels_posts_20260423_233414.sqlite`: **4 435 001** постов из **103** каналов, 2015-12-28 … 2026-04-24. Раздаётся сжатой zstd (≈1 ГБ, ссылка – в README, раздел «Получение данных»); распаковать в корень репозитория: `zstd -d <архив>.sqlite.zst`.

Сырые выгрузки `inputs/*.jsonl` (18 ГБ) не распространяются – их содержимое полностью представлено в базе. Скрипт `./reproduce.sh import` (`import_patriot_data_to_sqlite.py`) сохранён как документация происхождения базы для держателей исходных выгрузок.

## Конвейер

```
SQLite (4.44M постов)
   │ filters (CPU, детерминированно)
   ├─► outputs/president_putin_selection_v2(.jsonl → _for_model.jsonl)   207 776
   └─► outputs/executors_selection_…(.jsonl → _for_model.jsonl)          85 992
   │ topics (GPU): USER-bge-m3 → UMAP(seed=42) → HDBSCAN → BERTopic
   ├─► putin_v2:   docs_with_topics.parquet (161 599 док.), topic_info.csv
   └─► executors:  docs_with_topics.parquet (65 698 док.),  topic_info.csv
   │ sentiment (GPU): ансамбль = 2 классификатора + zero-shot GeRaCl
   │   (а) 2-модельный прогон исполнителей → *.parquet.bak_2model (для sensitivity)
   │   (б) run_geracl_zeroshot.py (env geracl) → zeroshot_geracl_probs.npy
   │   (в) --recompute-sentiment-only --zero-shot-precomputed → финальный parquet
   │   (г) финальный рендер: президент с --sentiment-exclude-mixed; эволюция слов тем
   │ stage3 (CPU, детерминированно)
   ├─► aggregates_sentiment_before_after.csv, stage3_summary.json, fig_negative_*
   └─► sensitivity_2model/ (2-модельный ансамбль, проверка устойчивости вывода)
   │ package
   └─► results/ (рисунки 01–06 + данные + README)
```

Этапы и ориентировочное время (4×GPU):

| Команда | Что делает | Время |
|---|---|---|
| `./reproduce.sh filters` | 4 фильтра корпусов из SQLite + подготовка входов | ~1.5–2 ч CPU |
| `./reproduce.sh topics` | эмбеддинги + BERTopic по двум корпусам | ~1.5–2.5 ч |
| `./reproduce.sh sentiment` | резервный двухмодельный прогон, GeRaCl zero-shot, финальный ансамбль, отрисовка графиков | ~3–5 ч |
| `./reproduce.sh stage3` | агрегаты до/после + sensitivity | ~10 мин |
| `./reproduce.sh package` | сборка `results/` | ~1 мин |
| `./reproduce.sh verify` | сверка с эталоном | ~3 сек |

`reproduce.sh` фиксирует точные команды всех этапов (включая неочевидные параметры: `--min-topic-size 50` у исполнителей, `--extra-stop-terms` под каждый корпус, `--exclude-footer-urls` в stage3).

## Замороженные артефакты (не генерируются скриптами)

Три входа создавались LLM-ассистированной разметкой/аудитом и зафиксированы в репозитории как данные. Перепрогон LLM дал бы немного другой результат, поэтому они заморожены – это сознательное решение для воспроизводимости:

| Файл | Что это | Контроль в verify |
|---|---|---|
| `outputs/rkn_register_audit/footer_crosspromo_rkn_urls.txt` | список URL постов, где «РОСКОМНАДЗОР» – рекламная подпись одноимённого канала в конце поста, ошибочно совпадавшая с термином «роскомнадзор»; подаётся в stage3 через `--exclude-footer-urls` | число строк |
| `outputs/blame_coding/blame_coding_results.csv` | LLM-кодирование «направления вины» (методология и валидация: `docs/llm_validation_methodology.md`) | число строк |
| `outputs/rkn_register_audit/*.md` | аналитические записки аудита | входят в `results/filelist` |

## Детерминизм: что совпадёт бит-в-бит, а что – с допуском

| Слой | Гарантия | Почему |
|---|---|---|
| filters, подготовка входов | бит-в-бит | чистый CPU-код, без случайности; verify сравнивает все счётчики триггеров/исключений точно |
| BERTopic при имеющемся кэше эмбеддингов (`*.npy` в out-dir) | бит-в-бит | UMAP с `random_state=42`, HDBSCAN детерминирован |
| эмбеддинги/тональность с нуля на GPU | float-шум | недетерминированные CUDA-ядра и шардирование по GPU; единичные документы у границы решения могут сменить класс |
| число тем при пересчёте эмбеддингов с нуля | ±несколько тем | сдвиг границ кластеров; проверка уровня WARN (допуск 30%) |
| доли тональности, метрики до/после | допуск 0.01–0.015 | усреднение по 65–160 тыс. док. гасит единичные расхождения |
| stage3, package при готовых parquet | бит-в-бит | детерминированный CPU-код; доверительные интервалы с фиксированным зерном случайности |

Практические режимы:
- **Режим А (проверка сданных артефактов)**: ничего не пересчитывать, сразу `./reproduce.sh verify` – подтверждает, что артефакты согласованы с эталоном.
- **Режим Б (перепрогон с кэшами)**: `./reproduce.sh all` поверх существующих каталогов – тяжёлые шаги используют кэш (parquet/npy), результат идентичен.
- **Режим В (с чистого листа)**: удалить `topic_model_outputs_dynamic_*` и `outputs/*selection*` и запустить `./reproduce.sh all` – все «точные» проверки обязаны сойтись, GPU-зависимые сходятся в пределах допусков.

## Автопроверка

`verify_reproduction.py` сверяет 28 контрольных точек: счётчики фильтров (включая полный словарь триггеров корпуса президента), число строк и mixed-постов в parquet, доли тональности, размеры агрегатов A/B2/C/GOV/PARL/AGENCIES, доли негатива до/после по каждому агрегату в обоих сценариях, число строк sensitivity-анализа, полный список файлов `results/`, замороженные артефакты.

Эталон снят с фактически сданных артефактов командой `python verify_reproduction.py --write-expected` (файл `reproduce_expected.json`).

Текущий статус: **PASS=28 WARN=0 FAIL=0**.
